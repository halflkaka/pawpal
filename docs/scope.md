# Scope & Guardrails

What is actively being worked on, what is deferred, and what to avoid investing in prematurely. Consult this before proposing or starting new work.

---

## Currently In Scope

Based on ROADMAP.md phases 1–3, which are the active focus:

- Feed and post surfaces — loading, creation, images, likes, comments
- Follow/unfollow and social graph
- Discovery / explore screen
- Profile and pet management
- **1:1 chat (text MVP)** — promoted out of deferred in #45. Migration 016 backs `conversations` + `messages`; `ChatService` owns inbox/thread loads + optimistic send. Scope limited to text-only DMs between two users. Entry points shipped in #46 (pet profile pill + follow-list shortcut) and #47 (`+` → `ComposeNewChatSheet` inside 聊天).
- **Onboarding** — promoted out of Phase 6 deferral in #47. `OnboardingView` full-screen gate forces first-pet creation before `MainTabView` renders its tab bar. Scope limited to pet creation; no tour / tooltip layer.
- **Stories (image-only MVP)** — shipped in #48. Migration 018 backs `stories`; `StoryService.shared` owns load/post/delete; `StoryComposerView` + `StoryViewerView` drive the UX. Scope: per-pet, 24h ephemeral, image-only on the client (schema accepts video for a later PR).
- **Passive virtual-pet decay** — shipped in #48. `VirtualPetStateStore.decayedState` applies hunger -3/hr, energy +2/hr, mood -1/hr on every read; `applyAction` stacks tap deltas on top of the decayed baseline. No scheduled job required — decay is a read-side transform.
- **Pet milestones (birthday + memory loop)** — shipped 2026-04-18. Birthday card pinned at top of FeedView when any owned pet has `pets.birthday` matching today. Memory card ("X年前的今天") on FeedView for posts whose month-day matches today and year < this year. "即将到来的纪念日" rail on PetProfileView. Composer prefill API on `CreatePostView` carries Chinese caption + pre-selected pet ID from any milestone tap. Backed by new `MilestonesService` (stateless, derived) and `PostsService.loadMemoryPosts(forUser:)` (server-side filtered fetch). Push notifications for milestone day-of land in v1.5 of the push pipeline.
- **Push notifications v1** — shipped 2026-04-18. `device_tokens` + `notifications` tables (migration 022) with RLS. AFTER INSERT triggers on `likes`, `comments`, `follows` call `queue_notification` which inserts the row and fires `pg_net.http_post` at the `dispatch-notification` edge function (Deno, ES256 APNs JWT, direct APNs — no FCM). iOS: new `PushService.swift`, `AppDelegate.swift`, `DeepLinkRouter.swift`; `@UIApplicationDelegateAdaptor` on the App struct; priming sheet shown after `OnboardingView` saves the first pet; token upsert/delete hooked into `AuthManager.signIn/register/restoreSession/signOut`; `pawpal://` URL scheme registered in `Info.plist`; `.onOpenURL` on `ContentView` routes external links. Direction doc: `docs/sessions/2026-04-18-pm-push-notifications.md`. v1.5 (milestone day-of, playdate reminders, chat DM pushes) reuses the same pipeline. APNs delivery gated on user-owned Apple Developer Program enrollment; code is complete and ships safely pre-enrollment (upsert failures logged + swallowed).
- **Local notifications stopgap** — shipped 2026-04-18. Device-scheduled milestone day-of reminders (birthdays only, v1) via `UNCalendarNotificationTrigger`. New `LocalNotificationsService.shared` owns schedule/cancel; identifier prefix `pawpal.milestone.birthday.<pet_uuid>`, fires 09:00 local on birthday month-day, repeats yearly. Rescheduled on `petsService.pets` change (diffed on `(id, birthday)` tuples to avoid `.petDidUpdate`-driven thrash) and scenePhase → active (catches permission flips in Settings). Cancelled on signOut. Tap routes through the same `AppDelegate.userNotificationCenter(_:didReceive:)` API as APNs, with new `DeepLinkRouter.Route.pet(UUID)` case landing on `PetProfileView` (Me tab for owned pets, Discover otherwise). Permission reuses the existing priming sheet — no separate prompt. Direction doc: `docs/sessions/2026-04-18-pm-local-notifications-stopgap.md`. Chosen as the long-term owner of device-schedulable events per `docs/decisions.md`; will not be superseded by APNs v1.5 birthday pushes.
- **Playdate scheduling MVP** — shipped 2026-04-18. Migration 023 adds `pets.open_to_playdates` (opt-in, default false) and the `playdates` table (proposer/invitee pet ids + denormalised user ids + scheduled_at + location + message + status ∈ proposed/accepted/declined/cancelled/completed) with airtight RLS, a BEFORE INSERT trigger enforcing the open-to-playdates gate, and an AFTER INSERT notify trigger that reuses migration 022's `queue_notification` with type `playdate_invited`. Edge function gains a `playdate_invited` branch with a `formatRelativeZh` helper. iOS: `PlaydateService.shared` owns inbox/outbox/detail loads, optimistic propose/accept/decline/cancel/markCompleted, and a `.playdateDidChange` broadcast; `RemotePlaydate` model; `LocalNotificationsService.schedulePlaydateReminders(for:otherPetNameByID:)` schedules three local reminders per accepted playdate (T-24h / T-1h / T+2h, identifier prefix `pawpal.playdate.`); `DeepLinkRouter.Route.playdate(UUID)`; 约遛弯 entry-point pill on `PetProfileView` gated on viewer-owns-a-pet + target-pet-open + not-self; one-time `PlaydateSafetyInterstitialView` (persisted via `UserDefaults "pawpal.playdate.safety.seen"`); `PlaydateComposerSheet` with `MKLocalSearchCompleter` location autocomplete via new `LocationCompleter`; `PlaydateDetailView` with status-aware Accept/Decline/Cancel; pinned `PlaydateRequestCard` + `PlaydateCountdownCard` rows on FeedView (48h unanswered invites + 4h upcoming accepted); `PostPlaydatePromptSheet` fires after completion and prefills `CreatePostView` via extended `ComposerPrefill(pets:)`; 开启遛弯邀请 toggle added to `ProfilePetEditorSheet`. Direction doc: `docs/sessions/2026-04-18-pm-direction-playdates.md`. Execution spec: `docs/sessions/2026-04-18-pm-playdates-mvp-execution.md`.
- **External share-out (微信 / 小红书)** — shipped 2026-04-18. New `ShareLinkBuilder` singleton centralises URL + Chinese message construction (`pawpal://post/<uuid>`, `pawpal://pet/<uuid>`, `pawpal://u/<slug>`). `ShareLink` affordances on `PostDetailView` + `PetProfileView` toolbars; `ProfileView.shareURLForSelf` refactored to delegate. `DeepLinkRouter` gained a `pawpal://u/` → profile alias. Handle-only profile round-trip (slug not a UUID) is noted as a follow-up — the share artefact is valid regardless.
- **Breed / city cohort surfaces** — shipped 2026-04-19. Breed + city pills on `PetProfileView` push a new paginated `PetCohortView` listing every pet in that cohort (24-row pages, `created_at desc`, pull-to-refresh). Discover's "与 X 相似的毛孩子" + "X 的毛孩子" rails gain a "查看全部" header link that routes to the same cohort view (breed-first for similar when available; city for nearby). Backed by new `PetsService.fetchPetsByBreed` / `fetchPetsByCity` with offset pagination. No schema change — breed / city remain string columns on `pets`.
- **Instrumentation (D7 retention, posts/DAU, sessions/week)** — shipped 2026-04-19. First-party event log only — no Firebase Analytics, no Mixpanel, no Amplitude, no Segment, no third-party SDK. Migration 025 adds `public.events` (uuid id, nullable `user_id` FK → profiles, `kind text`, `properties jsonb`, `client_at` + `server_at` timestamps) with airtight INSERT RLS (`user_id is null OR auth.uid() = user_id`) and no SELECT policy — analytics queries run server-side as `service_role`. `AnalyticsService.shared` is the single call-site fire-and-forget logger: `log(_:properties:)` dispatches a detached `Task`, reads the current session id off the shared Supabase client, and swallows any insert failure with a `[Analytics] … 失败` console line. `session_start` is debounced client-side to at most one per 30 minutes via `logSessionStart()`. Thirteen event kinds emitted in this PR (all additive — new kinds do NOT require a schema migration): `app_open` (App struct init), `session_start` (scenePhase → active + signIn + signUp), `sign_in` + `sign_up` (AuthManager success paths, `method = "password"`), `post_create` (PostsService success, `image_count` + `has_caption` dims), `story_post` + `story_view` (StoryService success paths, `story_id` for view), `playdate_proposed` + `playdate_accepted` (PlaydateService success paths), `share_tap` (ShareLink `.simultaneousGesture` on post / pet / profile surfaces), `follow` (FollowService success path, `target_user_id` dim), `like` + `comment` (PostsService success paths — insert only, NOT on unlike / delete, `post_id` dim). No PII beyond `user_id` + dimensional metadata; no device id, no advertising id, no IP (already logged by Postgres connection), no precise location, no user-authored text content. `TODO(analytics-opt-out)` seam reserved at the top of `AnalyticsService.swift` for a v1.5 opt-out UI + iOS ATT-style prompt if we ever ship in markets that require it. See `docs/decisions.md` → "First-party event log; no third-party analytics SDK".
- **Story view counts / seen-by** — shipped 2026-04-18. Migration 024 adds the `story_views` table — primary key `(story_id, viewer_pet_id)`, denormalised `viewer_user_id`, `viewed_at timestamptz default now()`, `ON DELETE CASCADE` on both FKs so viewer rows clean up with the parent story or pet. RLS is airtight: SELECT + DELETE gated on the story's owner via a subquery against `stories`; INSERT gated on `auth.uid() = viewer_user_id` with a SECURITY DEFINER BEFORE INSERT trigger (`story_views_gate_pet_ownership`) enforcing the pet-belongs-to-caller check — mirrors the `playdates_gate_invitee_open` pattern from migration 023. No UPDATE policy (rows are immutable); no client-side DELETE for viewers (ghost mode is explicitly deferred). iOS: new `RemoteStoryView` model with composite-PK synthetic `id`; `StoryService.shared` gains `recordView(storyID:viewerPetID:)` (upsert-ignoreDuplicates on `story_id,viewer_pet_id`, silent on RLS rejection), `viewerCount(storyID:)` (head-count query, 0 on error), `viewers(storyID:)` (embeds `viewer_pet:pets!viewer_pet_id(*)`, sorted `viewed_at desc`). `StoryViewerView` silently records one view per story per session (deduped via `recordedViewIDs: Set<UUID>`, viewing-as-pet defaults to `PetsService.shared.pets.first`); owners get a new "眼睛 icon + N 位看过" chip above the safe-area inset that presents the new `StoryViewersSheet.swift` (pet avatar + name + relative timestamp rows, pull-to-refresh, taps push `PetProfileView`). Chinese-only copy; design tokens from `PawPalTheme` throughout. Stories RLS policies from migration 018 remain untouched.

## Deferred — Do Not Invest Here Yet

- **Chat realtime, stickers, reactions, unread, presence** — #45 shipped the text-DM MVP, but typing indicators, online dots, sticker tray, per-message reactions, and unread badges are intentionally out of scope. The `messages` schema doesn't carry `last_read_at` / `read_by` yet; grow the schema before adding UI.
- **Push notifications v1.5** — chat DM pushes (`chat_message`) and memory-loop pushes (`memory_today`) still need `buildPayload` branches. As of playdates MVP (2026-04-18) the `playdate_invited` branch has landed; the `playdate_t_minus_24h` / `playdate_t_minus_1h` / `playdate_t_plus_2h` reminders are intentionally owned by the **local** notifications path (see `docs/decisions.md` "Local notifications own device-schedulable events") and will not be added to the edge function. Note: milestone day-of (`birthday_today`) stays out of the APNs path — the local stopgap is canonical.
- **Playdates follow-ups** — the 2026-04-18 MVP shipped with explicit follow-up seams: Discover rail / map mode for finding nearby open pets; a server-side completed sweeper (cron'd RPC that flips `scheduled_at < now - 2h AND status = 'accepted'` to `completed` so every device observes the transition even if the app is cold); optional realtime subscription for cross-device cancel updates; group playdates (>2 pets). Weekly-repeat playdates landed 2026-04-19 (migration 027 + `series_id` / `series_sequence` columns + composer toggle + detail-view "取消整个系列" menu). **Group playdates landed 2026-04-19** (migration 028 + `playdate_participants` junction + composer multi-select up to 2 invitees + detail-view participant scroll with per-pet status + three SECURITY DEFINER RPCs). The remaining follow-ups (Discover rail / map mode, realtime subscription) do not ship in MVP+0. Group-chat layer for >3 pets stays out of scope — see `docs/decisions.md` "Group playdates — junction table, not array columns".
- **Push notification polish** — quiet hours, per-type mute, in-app notification center, rich push (image attachments via Notification Service Extension), VoIP for playdate T-1h, server-side unread count / badge coalescing. All deferred per the 2026-04-18 push direction doc; seams are reserved.
- **Pet-specific follow** — Phase 4. Current follow graph is user-to-user only.
- **App Store / TestFlight prep** — Phase 6. Not a current concern.
- **Feed algorithm** — Phase 6. Feed is chronological for now; do not add ranking logic.
- **Video stories** — #48 shipped image-only stories. Migration 018 and `StoryService.postStory` both accept `media_type = 'video'`, but the composer's `PhotosPicker` is locked to `.images` and the viewer uses `AsyncImage` (still). Wiring `AVPlayerLayer` + transcode comes in a later PR; search `TODO(video)` for seams.
- **Realtime stories rail + chat unread/presence** — still deferred. The rail refreshes on pull-to-refresh + after publish, not via subscriptions.
- **Pet care / health logging** (vaccines, weight log, vet reminders, food log) — explicitly OUT of scope per the 2026-04-18 PM direction reset. Reasoning: utility silo that competes with dedicated pet-product apps (小佩, Fi) on their turf, doesn't strengthen PawPal's pet-first social core. Do not propose this without reopening the question with the user.
- **Extended milestone kinds** (gotcha day, account anniversary, first playdate, "first" milestones) — MVP ships birthday + memory only. `MilestonesService.MilestoneKind` is intentionally an open enum so adding kinds is additive. Don't ship new kinds before push lands — they read as dead affordances without notification support.

- Check `ROADMAP.md` and this file before starting any new feature work
- If a proposed change touches a deferred area, flag it and confirm before proceeding
- Prefer depth over breadth — finish and polish existing surfaces before adding new ones
